# TOGBankClassic — Claude Instructions

WoW Classic Era addon (Interface 11508 only — no retail/Wrath/Cata variants). Lua 5.1, external Ace3 (AceAddon-3.0, AceComm-3.0, AceConsole-3.0, AceDB-3.0, AceEvent-3.0, AceGUI-3.0, AceSerializer-3.0, AceTimer-3.0), LibStub, LibDataBroker-1.1, LibDBIcon-1.0, AceCommQueue-1.0. Source layout: `Core.lua` + `Modules/*.lua` + `Modules/UI/*.lua` + `Libs/` (only non-Ace3 libs vendored).

## Always

- **Module namespace** — every module file declares one top-level global `TOGBankClassic_<Name>` table (e.g. `TOGBankClassic_Bank`, `TOGBankClassic_Database`, `TOGBankClassic_Guild`, `TOGBankClassic_Output`). UI sub-modules use `TOGBankClassic_UI_<Window>` (e.g. `TOGBankClassic_UI_Inventory`, `TOGBankClassic_UI_Requests`). Never use `local addon`/`local ns` wrappers — modules reference each other by global name. New module files must be added to `TOGBankClassic.toc` in dependency order (Constants → Output → Performance → DeltaComms → Bank → Chat → Database → … → UI → UI sub-modules → Tests → Core).
- **All comms go through `TOGBankClassic_Core`** — never call `AceComm:SendCommMessage` directly. Use `TOGBankClassic_Core:SendCommMessage(prefix, text, distribution, target, prio, cb, cbArg)` (raid auto-suppressed, AceCommQueue-wrapped, debug-logged) or `TOGBankClassic_Core:SendWhisper(prefix, text, target, prio, cb, cbArg)` (adds online check + same-realm name stripping). Every new prefix must be registered in `COMM_PREFIX_DESCRIPTIONS` in [Modules/Constants.lua](Modules/Constants.lua) or the SEND log will print `(Unknown)`. The current prefix inventory and typical sync flow are documented in the header comment of [Modules/Chat.lua](Modules/Chat.lua) — read it before adding a prefix.
- **Serialized payloads use checksum framing** — wrap outgoing tables with `TOGBankClassic_Core:SerializeWithChecksum(data)` and unwrap with `TOGBankClassic_Core:DeserializeWithChecksum(message, ctx)`. Wire format is `<AceSerialized>\030<checksum>\031END`; the stop-marker + CRC pair distinguishes truncation from bit-corruption. Pass `ctx = { sender = sender, prefix = prefix, distribution = dist }` to get useful INTEGRITY-MISMATCH diagnostics on the receive side. Old pre-checksum messages still deserialize (graceful fallback).
- **AceDB scopes** — only two are used. `db.global` holds debug toggles (`debugCategories`, `debugTags`, `showUncategorizedDebug`). `db.faction[guildName]` holds all per-guild state: `roster`, `alts`, `requests`, `requestsTombstones`, `settings`, `guildProtocolVersions`, `deltaMetrics`, `deltaErrors`. Use the tightest appropriate scope. SavedVariables names: `TOGBankClassicDB` (main), `TOGBankClassicIconDB` (minimap), `TOGBankClassicOptionDB` (legacy options), `TOGBankClassicDB_DebugLog`, `TOGBankClassic_PerfMetrics`, `TOGBankClassic_PerfEnabled`, `TOGBankClassic_DebugLogEnabled` — all declared in [TOGBankClassic.toc](TOGBankClassic.toc).
- **Logging — always use the category+tag form** — `TOGBankClassic_Output:Debug("CATEGORY", "TAG", fmt, ...)` (preferred) or `TOGBankClassic_Output:Debug("CATEGORY", fmt, ...)` (no tag). Never `print()` or raw `DEFAULT_CHAT_FRAME:AddMessage` for debug output. Categories live in `DEBUG_CATEGORY` ([Modules/Constants.lua](Modules/Constants.lua)) and must also be initialised to `false` in `Database:Init()`'s `debugCategories` default ([Modules/Database.lua](Modules/Database.lua)) and given a display row in `CATEGORY_META` ([Modules/Options.lua](Modules/Options.lua)) — all three must stay in sync. Tags are opt-out (new tags auto-show); add a description to `DEBUG_TAGS[CATEGORY][TAG]` in Constants so users can see what to suppress. User-visible (non-debug) output uses `Output:Response`, `Output:Info`, `Output:Warn`, `Output:Error`.
- **AceGUI usage** — `TOGBankClassic_UI` *is* the AceGUI-3.0 library (`TOGBankClassic_UI = LibStub("AceGUI-3.0")` in [Modules/UI.lua](Modules/UI.lua)), so `TOGBankClassic_UI:Create(widgetType)` is the AceGUI factory. Per-window code lives in `TOGBankClassic_UI_<Window>` sub-modules. Use AceGUI widget methods (`widget:SetText(...)`), only drop to `widget.frame` when a WoW API has no AceGUI equivalent. Persistent windows (e.g. Inventory, Requests) are stored on their sub-module table and hidden on close, not released — refresh by re-populating in place. Apply the standard thin border via `TOGBankClassic_UI:ApplyThinBorder(widget)`.
- **Guild roster lookups** — `TOGBankClassic_Guild.memberRoster` is the authoritative full-roster cache with `isOnline` state (keyed by `Name-Realm`); prefer it over the legacy `onlineMembers` table or iterating `GetGuildRosterInfo` directly. Use `TOGBankClassic_Guild:IsPlayerOnline(name)` and `TOGBankClassic_Guild:IsBank(name)` rather than rolling your own checks. Banker detection is cached in `banksCache` — invalidate it (not re-iterate) when roster changes.
- **Banker identity is note-based** — a guild member is a banker when its public/officer note contains the substring `gbank` (`note:find("gbank", 1, true)`). VIEWBANK-001 adds a *view-only* flag on top of that: a banker whose note also carries a view-only marker (`viewonly`, `view-only`, `readonly`, `read-only`, or the compact `gbankro`; see `VIEW_ONLY_MARKERS` in [Modules/Guild.lua](Modules/Guild.lua)) is visible everywhere but not requestable. Use `TOGBankClassic_Guild:IsViewOnlyBank(name)` (O(1) via the `viewOnly` flag on `memberRoster`, kept in sync alongside `isBank` in `RefreshOnlineCache`/`RebuildBankerRoster`). The request hard-gate lives in `Guild:AddRequest`; never create a request bypassing it.
- **Request lifecycle goes through `TOGBankClassic_Guild` / RequestLog wire helpers** — never mutate `db.faction[name].requests` or `requestsTombstones` directly from UI code; route through the public methods so version counters, tombstone TTL, and the `togbank-rm` ALERT-priority mutation broadcast all fire correctly. The compact positional wire formats `togbank-ri` (index, `RI_VERSION = 1`) and `togbank-rd2` (records, `RD2_VERSION = 1`) are documented at the top of [Modules/RequestLog.lua](Modules/RequestLog.lua). When adding optional fields to the record, append to the end and tolerate `false`/absent on receive (see the `itemID` field at slot 12 added for REQ-001 same-name disambiguation).
- **Same-name item variants** — names from `GetItemInfo` are not unique (e.g. class-specific Punctured Voodoo Doll variants). Match on `request.itemID` (numeric) when available, fall back to `request.item` (name) only when `itemID` is absent (old clients). This applies in bag scanning, mail fulfillment detection, item highlighting, and tooltip lookups. See REQ-001/REQ-002 in [CHANGELOG.md](CHANGELOG.md) for the full breakdown.
- **Lua globals** — [.luarc.json](.luarc.json) lists every WoW API global the Lua language server needs to recognize. If you add a new WoW API call and the LS warns `undefined-global`, append to `diagnostics.globals` in that file. `runtime.version` is pinned to `Lua 5.1`; `need-check-nil` is globally disabled.
- **Markdown lint** — [.markdownlint.json](.markdownlint.json) configures rules for all `.md` files; MD047 (single trailing newline) is enforced — always finish `.md` files with one trailing newline.
- **Fix lint/compile errors, don't suppress them.** The point of linting is to surface real issues; suppressing a warning hides the signal instead of addressing it. Apply fixes at the right scope:
  - **`undefined-global`** (WoW API names, addon globals like `TOGBankClassic_ItemDB`) → add to `diagnostics.globals` in [.luarc.json](.luarc.json). The fix is to teach the LS that the symbol exists.
  - **`undefined-field` on a typed table** (e.g. `builderState.items`) → add a `---@class TableName` with `---@field` annotations to declare the shape. This is type info, **not** a suppression — it lets the LS catch field-name typos (`builderState.itmes`) at lint time.
  - **`unused-function`** (dead code) → delete the function. If it's a dispatch table entry or genuinely planned-but-not-yet-used, document why with a comment so the next pass doesn't re-flag it.
  - **`lowercase-global`** (missing `local` keyword) → add `local`. If it really must be global, give it the addon's namespace prefix.
  - **`redundant-parameter`** on AceComm/AceConsole methods that the LS doesn't fully understand → `---@diagnostic disable-next-line` on that specific call, with a comment explaining why.
  - **Genuine noise from dynamic Lua + AceGUI patterns** (`unused-local`, `unused-vararg`, `redefined-local`, `need-check-nil`) → these are stylistic preferences that aren't actionable signals in this codebase (AceGUI callbacks pass widgets we often don't use, `function(self)` shadowing inside scripts is universal, nil checks are everywhere). Disabled globally in `.luarc.json`'s `diagnostics.disable` so they don't generate noise. **New code should still use `_` for genuinely-unused params** so the intent is explicit.
  - **`---@diagnostic disable-next-line: code-name`** is the last resort for a single line that legitimately needs a one-off suppression — never a substitute for either of the above.
- **Comments must be accurate** — only explain the non-obvious "why", not what the code already says. Never write aspirational comments describing intended rather than actual behaviour. Update stale comments in any block you edit. Follow the established pattern of inline ticket IDs (`REQ-001`, `MAIL-014`, `P2P-032`, `PERF-012`, `SYNC-001`, etc.) when a comment is the audit trail for a specific bug — keeps the in-code reference greppable against `CHANGELOG.md`.
- **Look at existing files first** — established patterns live in [Modules/Bank.lua](Modules/Bank.lua) (inventory scanning), [Modules/Guild.lua](Modules/Guild.lua) (roster + P2P coordination), [Modules/RequestLog.lua](Modules/RequestLog.lua) (request mutations + wire formats), [Modules/Chat.lua](Modules/Chat.lua) (slash commands + comms inventory), and [Modules/UI/Requests.lua](Modules/UI/Requests.lua) (canonical AceGUI window with persistent layout, tooltips, paginated rows). Check them before inventing a solution.
- **Scratch files use `tmpclaude-` prefix** — established convention for any throwaway Lua/MD files written during investigation. The `.pkgmeta` ignore list catches them defensively but the naming convention is the primary safeguard against shipping them by accident.
- **📋 Documentation updates (REQUIRED CHECKLIST)** — after implementing any feature or fix, ask the user which version it belongs to, then update **BOTH** of these files:
  1. **[CHANGELOG.md](CHANGELOG.md)** — prepend a new version section in the established format (`## [vX.Y.Z] (YYYY-MM-DD) - <Title>` with `### New Features` / `### Bug Fixes` / `### Improvements` / `### Internal` subsections). Use ticket-ID prefixes (`TOOLTIP-001`, `MAIL-014`, etc.) for bug entries so the ID is greppable from inline code comments back to the changelog. Include file paths in the form `Modules/Foo.lua`.
  2. **[docs/Curseforge_Description.html](docs/Curseforge_Description.html)** — add an `<h3>` block under "What's New" with end-user-friendly language. **No** code identifiers, function names, file names, state constants, or internal architecture terms (sessions, handshakes, anchors, hash-list, AceCommQueue, P2P, etc.). Ticket IDs (e.g. `TOOLTIP-001:`) are kept as a short prefix users can quote in bug reports — that's the only acceptable code-side reference. Describe the user-visible change: what they see, do, or no longer experience.
  - Also update any relevant file in `docs/` (e.g. `FEATURE_IMPROVEMENTS.md`, `ORDER_FULFILLMENT_LOGIC.md`, `DELTA_BUGS.md`, design docs) if the change affects something documented there.
  - `README.txt` is the player-facing manual and only needs updating when commands, setup, or major features change — not for every fix.
- **HTML patch notes must be end-user friendly** — `docs/Curseforge_Description.html` is read by players, not developers. Each bullet should describe the user-visible change in plain language: what they see, do, or no longer experience. Keep technical depth (function names, file paths, root-cause analysis, ticket IDs beyond a short prefix) in `CHANGELOG.md` and the `docs/*.md` design notes — those are the dev-facing records.
- **No version replication, no local build script** — unlike most Ace3 addons, TOGBankClassic ships Classic Era only (Interface 11508 in [TOGBankClassic.toc](TOGBankClassic.toc), no `_BCC`/`_Wrath`/`_Mainline` TOC variants, no version-replication script). Builds are produced exclusively by the BigWigs CurseForge packager via [.pkgmeta](.pkgmeta) (Ace3 + VersionCheck pulled as required-dependencies, never embedded; the `@project-version@` token in the TOC is substituted by the packager). No local `package.bat` / `dist/` — those were retired when the project moved to the CurseForge auto-build pipeline. Do not commit anything to a `_retail_`/`_classic_`/`_anniversary_` path — there are none.
- **Release tag format** — git tags for this addon **must** be `TOGBankClassic-vX.Y.Z` (e.g. `TOGBankClassic-v1.2.3`). The CurseForge packager keys its builds off the tag, so the `TOGBankClassic-` prefix and the `v` before the semantic version are both required — never tag a plain `v1.2.3` or `1.2.3`.
